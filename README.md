# Guitar Journey

吉他自学项目需求大纲 v1.0

## 1. 项目定位

- 项目类型：网页应用
- 项目方向：吉他训练课程系统，附带学习记录
- 使用对象：前期自己使用，后期可以公开
- 项目风格：音乐风
- 主要乐器：钢弦民谣吉他
- 目标：从基础恢复开始，逐步学习弹唱、乐理、和弦原理、节奏型、指弹、扒谱、创作歌曲。

## 2. 核心学习目标

用户最终希望掌握：

1. 弹唱
2. 指弹
3. 即兴伴奏
4. 扒歌
5. 作曲
6. 和弦组成原理
7. 和弦走向分析
8. 转调、变调夹、调式分析

当前优先恢复：

1. 和弦转换
2. 手指灵活度
3. 和弦组成原理
4. 基础乐理习题

## 3. 学习推进方式

项目不按固定日期推进，而是按学习进度推进。

每个关卡包含：

1. 学习新知识
2. 做知识题
3. 练习吉他
4. 系统评分
5. 及格后进入下一关

如果某天没有练习，不影响整体进度，只是不生成练习记录。

## 4. 每周练习时间

- 周一到周五：每天约 20 分钟
- 周六：约 1 小时
- 周日：约 1 小时

练习内容不固定按日期安排，而是根据当前关卡和完成度自动安排。

## 5. MVP 第一阶段

第一阶段只做最小可用版本：

- 练习记录
- 曲谱练习页面

包含：

1. 曲谱列表
2. 曲谱详情
3. 用户看着曲谱练习
4. 记录练习时间
5. 记录曲目
6. 记录难点
7. 记录熟练度
8. 保存练习记录
9. 查看历史练习记录

暂时不做：

- 完整和弦库
- 评分系统
- 音频识别
- 相机识别

## 6. 后续功能模块

### 6.1 今日练习任务

显示当前应该学习或练习的内容。

包括：

1. 当前关卡
2. 今日建议练习
3. 当前歌曲
4. 当前和弦
5. 当前节奏型
6. 是否已打卡

### 6.2 和弦库

和弦库作为字典功能存在，但学习时不能一次性展示全部内容。

每个和弦包含：

1. 和弦名称
2. 按法
3. 构成音
4. 根音、三音、五音
5. 功能说明
6. 常见和弦走向
7. 不同把位按法
8. 相关练习

学习方式：

- 先解锁简单和弦，再逐步解锁复杂和弦。

### 6.3 节奏型库

包含：

1. 4/4 节奏型
2. 6/8 节奏型
3. 扫弦节奏
4. 分解和弦
5. 指弹基础节奏

### 6.4 歌曲库

用于记录自己练习的歌曲。

字段包括：

1. 歌名
2. 歌手
3. 调式
4. 变调夹位置
5. 使用和弦
6. 练习难度
7. 练习状态
8. 个人备注

注意：不要直接存储完整歌词、商业吉他谱、原书谱例，避免版权问题。可以存自己的练习备注和自己扒出来的和弦。

### 6.5 扒谱笔记

用户自己听歌判断和弦并记录。

包含：

1. 歌曲名称
2. 时间段
3. 自己判断的和弦
4. 旋律备注
5. 不确定点
6. 修改历史

### 6.6 乐理学习模块

从最基础开始：

1. 音名
2. 半音和全音
3. 音程
4. 三和弦
5. 七和弦
6. 顺阶和弦
7. 和弦走向
8. 调式
9. 转调
10. 变调夹

每个知识点需要配小习题。

### 6.7 指弹模块

先从简单分解和弦开始。

学习顺序：

1. 右手手指编号
2. 分解和弦
3. 简单指弹节奏
4. 低音 + 高音
5. 简单独奏曲

### 6.8 创作歌曲模块

用于记录原创内容。

包含：

1. 歌名
2. 灵感来源
3. 歌词草稿
4. 和弦进行
5. 旋律备注
6. Demo 版本
7. 修改历史

支持版本管理：

1. demo v1
2. demo v2
3. demo v3

## 7. 关于系统评分

第一阶段不做复杂评分。

后期可以考虑：

1. 使用浏览器麦克风录音
2. 分析节奏是否稳定
3. 分析音高是否接近
4. 给出简单评分

技术可行性：

- 音频评分可以逐步实现。
- 相机实时识别手型较难，需要图像识别模型，不建议第一阶段做。
- 第一阶段建议先做手动评分，例如熟练度 1 到 5 分。

## 8. 版权规则

项目可以参考自己的吉他书，但不能直接复制原书内容。

允许：

1. 总结知识点
2. 整理学习顺序
3. 写自己的理解
4. 写自己的练习记录
5. 记录自己扒出来的和弦

不建议：

1. 上传完整书籍内容到公开项目
2. 复制原书图片
3. 复制商业曲谱
4. 复制完整歌词
5. 公开发布未经授权的教学内容

## 9. 学习式开发规则

这个项目的开发过程优先让用户理解，而不是由 Codex 一次性全部做好。

默认流程：

1. Codex 说明一个很小功能点的目的。
2. Codex 只给这个小功能所需的必要代码。
3. 用户阅读代码。
4. 用户自己加中文备注。
5. Codex 检查备注是否正确。
6. 如果备注理解错了，Codex 必须直接指出哪里错，并用简单中文解释原因。
7. 用户自己手动写进项目。
8. 用户上传代码后，Codex 需要用中文生成 Commit 文本。
9. 每完成一个 Step，Codex 生成一份学习报告。
10. 学习报告完成后，Codex 再生成 Commit 内容。
11. 完成后再进入下一个小功能。

限制：

1. 每次只讲解并生成一个很小的功能点。
2. 不一次性生成完整模块代码。
3. 生成代码前，先说明这个小功能的目的。
4. 只给必要代码，不提前写后面 Step 的代码。
5. Codex 不直接替用户改项目代码，除非用户明确要求。
6. 代码要简单清楚，适合初学者理解。
7. 重要概念要边写边解释。
8. 如果用户问“为什么这样写”，必须从基础概念解释。
9. 如果用户写的备注有问题，不能只说“基本正确”，要指出具体问题。
10. 每个小功能的代码量要小，适合用户手动输入。
